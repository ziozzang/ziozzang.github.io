금융회사 망분리의 이해와 서버·서비스 구축 시 주의사항

망분리를 단순히 “인터넷 차단”이나 “물리 분리”로만 이해하면 클라우드, SaaS, 개발망, AI 연동 환경에서 판단을 그르치기 쉽다. 실무 판단의 핵심은 업무·데이터·접속경로·예외근거·보완통제를 함께 증명하는 것이다.

결론 요약

금융회사에서 망분리는 외부 위협이 내부 업무망·전산실·정보처리시스템으로 확산되는 것을 막기 위한 기본 통제다. 그러나 2024년 이후 정책 방향은 “무조건 차단”에서 “업무별 위험평가, 보완통제, 결과책임”으로 이동하고 있다.

핵심 해석
  • 망분리는 인터넷과 내부망을 끊어 놓는 장비 구성이 아니라, 전자금융거래 안전성과 정보기술부문 보호를 위한 접근·통신·자료이동 통제 체계다.
  • 클라우드 이용절차를 수행했다는 사정만으로 망분리 예외가 자동 승인되는 것은 아니다. 클라우드 이용, SaaS 예외, 연구·개발망 예외, AI 활용 특례는 각각 별도 요건과 보완통제를 확인해야 한다.
  • 서버·서비스 구축자는 “이 서버가 어느 망에 위치하는가”보다 “무슨 업무·데이터를 처리하고, 어떤 외부 접점이 있으며, 누가 어떤 경로로 운영하는가”를 먼저 문서화해야 한다.
  • 논리적 분리, VPC, 서브넷, 보안그룹, Kubernetes NetworkPolicy 등은 보완통제의 일부일 수 있지만, 그 자체가 금융권 망분리 충족 결론을 대체하지 않는다.
근거: 전자금융거래법 제21조 안전성 확보의무; 전자금융감독규정 제15조; 금융위원회, 「금융분야 망분리 개선 로드맵」, 2024.8.13, p.1~3, p.5~9; 금융보안원, 「내부 업무망 SaaS 보안가이드」, 2026.

서버·서비스 구축자의 5문장 원칙

  1. 서비스를 업무망, 인터넷망, 전산실, DMZ, 클라우드 테넌트, 연구·개발망 중 어디에 둘지 결정하기 전에 처리 업무와 데이터를 등급화한다.
  2. 외부통신망 접점, 원격관리 경로, API 호출 경로, 파일 반입·반출 경로를 아키텍처 다이어그램에 표시한다.
  3. 망간 통신이 있으면 “필요성, 최소 포트, 승인자, 기간, 로깅, 악성코드 검사, 사후 점검”을 남긴다.
  4. 클라우드와 SaaS는 계정·권한·로그·암호화·데이터 위치·위탁계약·출구전략을 함께 검토한다.
  5. 예외는 편의가 아니라 통제 책임의 증가다. 예외를 적용할수록 보안성 심의, CISO/경영진 보고, 운영 증적을 강화해야 한다.

법적 출발점: 안전성 확보의무와 전산실 보호

망분리 검토의 최상위 출발점은 전자금융거래법 제21조다. 이 조항은 금융회사 등이 전자금융거래가 안전하게 처리될 수 있도록 선량한 관리자로서 주의를 다하고, 금융위원회가 정하는 정보기술부문 기준을 준수해야 한다는 구조를 둔다.

전자금융거래법 제21조의 실무 의미

망분리는 제21조의 “안전성과 신뢰성 확보”를 구현하는 하위 통제다. 따라서 장비 도입 여부보다 전자금융거래 처리 안정성, 이용자 보호, 정보처리시스템 보호, 사고 예방이 실제로 달성되는지가 중요하다.

근거: 전자금융거래법 제21조 안전성의 확보의무, law.go.kr. 원문 요지: 금융회사 등은 전자금융거래의 안전성과 신뢰성 확보를 위해 금융위원회가 정하는 기준을 준수해야 함.

전자금융감독규정 제15조는 전산실 등에 관한 사항과 망분리 예외 판단의 직접 근거로 연결된다. 구축 실무에서는 제15조를 “전산실 출입통제” 정도로 좁게 보면 안 된다. 정보처리시스템 보호대책, 업무용 단말과 외부통신망 연결 통제, 전산자료 보호 및 접근통제, 예외 사유와 보완조치를 함께 보아야 한다.

망분리의 개념: 네트워크가 아니라 위험 전파 차단

금융회사에서 망분리는 인터넷 등 외부통신망에서 발생한 악성코드, 계정탈취, 공급망 침해, 자료 유출 위험이 내부 업무망과 중요 정보처리시스템으로 전파되지 않도록 통신 경로와 운영 경로를 제한하는 체계다.

좁은 이해
  • 인터넷 PC와 업무 PC를 분리한다.
  • 방화벽으로 내부망과 외부망을 나눈다.
  • 서버를 DMZ와 내부망에 나누어 둔다.
실무상 필요한 이해
  • 업무·데이터·사용자·관리자·API별 접속 경로를 통제한다.
  • 망간 자료이동과 원격운영 경로를 승인·검사·기록한다.
  • 예외를 적용할 때 보안성 심의와 보완통제를 설계한다.

물리분리와 논리분리

물리분리는 장비, 회선, 단말, 서버 영역을 물리적으로 분리하는 방식이다. 논리분리는 동일 물리 기반 위에서 VLAN, VPC, 서브넷, ACL, 보안그룹, 방화벽, 정책 엔진, 계정·권한 통제로 접근을 분리하는 방식이다. 금융권에서는 논리분리가 가능한 경우가 늘고 있지만, “논리적으로 나누었다”는 사실만으로 모든 망분리 의무가 충족되는 것은 아니다.

오해 금지

클라우드 VPC, Kubernetes namespace, NetworkPolicy, 보안그룹은 강력한 통제 수단이지만, 금융회사 업무망·전산실·외부통신망 예외 요건을 자동으로 충족시키는 일반 면책 수단이 아니다. 업무 중요도, 데이터, 외부 접점, 운영 계정, 로그, 사고 대응까지 함께 증명해야 한다.

근거: 전자금융감독규정 제15조 관련 문서 「전자금융감독규정 제15조 전산실 등에 관한 사항 및 망분리 예외」, https://www.law.go.kr/LSW/admRulLsInfoP.do?admRulSeq=2100000274812; 금융위원회 「금융분야 망분리 개선 로드맵」 p.8~9.

왜 해야 하는가: 사고 확산과 책임을 줄이는 장치

금융권 망분리는 대형 전산 장애와 해킹 사고 이후 강화된 규제 체계의 일부다. 금융위원회 로드맵은 금융권 망분리 도입 이후 약 10년이 경과했고, 클라우드·생성형 AI 확산에 따라 혁신과 보안의 균형을 다시 잡아야 한다고 설명한다.

적용 대상 판단: 서버가 아니라 업무·데이터·접점 기준

서버를 구축할 때 가장 먼저 할 일은 서버명을 정하는 것이 아니라 업무 분류표를 만드는 것이다. 동일한 웹서버라도 전자금융거래 처리, 인증, 고객정보 조회, 내부 업무지원, 홍보 페이지, 개발 테스트 여부에 따라 망분리와 보완통제 수준이 달라진다.

분류 순서

  1. 업무 중요도: 전자금융거래 처리, 인증, 계정관리, 결제, 거래 원장, 고객 상담, 내부 협업, 개발·테스트 중 무엇인지 구분한다.
  2. 데이터: 개인신용정보, 고유식별정보, 인증정보, 거래정보, 가명정보, 내부기밀, 공개정보를 구분한다.
  3. 접점: 인터넷 공개, 제휴기관 연계, SaaS, 클라우드 API, 내부 업무망, 전산실, 관리자 원격접속 경로를 표시한다.
  4. 운영자: 내부 직원, 외주 운영자, CSP 운영자, SaaS 관리자, 개발자, 보안관제 인력의 권한 범위를 분리한다.
  5. 예외: 망간 통신, 파일 반출, 외부 API 호출, SaaS 접속, 원격근무, 개발망 전송이 있는지 확인한다.

표준 아키텍처: 업무망·DMZ·전산실·클라우드의 역할

서버 구축 원칙: 기본값은 최소 연결·최소 권한

서버 구축자는 망분리를 인프라팀의 방화벽 업무로 넘기면 안 된다. 애플리케이션 포트, 배치 작업, 관리자 접속, 패치 경로, 로그 전송, 백업, 모니터링 에이전트, 외부 API 호출이 모두 망분리 판단에 영향을 준다.

서버 신규 구축 시 설계 기준

  • 서버 역할 단일화: 공개 웹, API 중계, 업무처리, DB, 배치, 관리자 도구를 가능한 한 분리한다.
  • 인바운드 최소화: 인터넷에서 직접 접근 가능한 서버는 DMZ 또는 공개 서브넷에 한정하고 내부 처리 서버로 직접 연결하지 않는다.
  • 아웃바운드 통제: 내부 서버의 인터넷 직접 접속을 금지하고, 패치·업데이트·외부 API는 프록시 또는 승인된 중계 경로를 사용한다.
  • 운영 접속 분리: 운영자는 업무 단말에서 서버로 직접 SSH/RDP하지 않고, Bastion, PAM, 세션기록, MFA를 거친다.
  • 파일 이동 통제: 배포 파일, 소스코드, 로그, 데이터 추출물은 승인·악성코드 검사·무결성 확인·반출 기록을 갖춘다.
  • 자동화 권한 제한: CI/CD 토큰, API Key, 서비스 계정은 용도별로 분리하고 만료·회전·비밀관리 체계를 적용한다.
자주 발생하는 설계 실패

“패치를 위해 임시 인터넷 오픈”, “장애 대응을 위해 운영자 PC 직접 접속”, “배치 서버에서 외부 API를 자유 호출”, “개발망에서 운영 DB 샘플 복사”는 모두 망분리 우회로가 될 수 있다. 임시 허용은 기간, 사유, 승인자, 정책 원복, 로그 검토가 없으면 상시 예외로 굳어진다.

클라우드 구축: 이용절차와 망분리 판단을 분리해서 보라

클라우드는 금융회사 전산실의 확장이 될 수도 있고, 외부 위탁·제3자 서비스가 될 수도 있다. 전자금융감독규정 제14조의2에 따른 클라우드 이용절차를 수행하더라도 제15조의 망분리 예외나 외부통신망 접점 통제가 자동으로 해결되는 것은 아니다.

클라우드 이용절차의 핵심 항목

금융위원회 로드맵은 클라우드 이용절차 예시로 이용업무 중요도 평가, CSP 평가, 업무연속성 계획, 안전성 확보조치, 계약 주요 기재사항을 제시한다. 이 절차는 클라우드 이용의 안정성을 평가하는 절차이지, 모든 내부망 외부 접속을 허용하는 절차가 아니다.

근거: 금융위원회 「금융분야 망분리 개선 로드맵」 p.9.

클라우드 VPC 설계 체크

  • 금융회사 전용 계정·구독·프로젝트를 업무별로 분리했다.
  • 인터넷 공개 서브넷, 애플리케이션 서브넷, DB 서브넷, 관리 서브넷을 분리했다.
  • 관리자 콘솔 접근은 SSO, MFA, 조건부 접근, 관리자 권한 승인 절차를 거친다.
  • 보안그룹·NACL·라우팅 테이블은 기본 차단 후 필요한 흐름만 허용한다.
  • NAT, 프록시, 패치 저장소, 외부 API Gateway를 통해 아웃바운드를 통제한다.
  • CloudTrail, Activity Log, Audit Log 등 관리행위 로그를 중앙 수집한다.
  • KMS/HSM, 고객관리형 키, 키 접근권한, 키 회전 정책을 정했다.
  • 백업·DR 리전, 데이터 위치, 국외 이전 여부, 삭제·반환 절차를 문서화했다.

SaaS 예외: 내부 업무망 접속 허용은 조건부다

2026년 시행세칙 개정으로 금융회사의 내부 업무망 SaaS 활용은 확대되었지만, “내부 업무망에서 외부 SaaS 접속이 항상 가능하다”는 의미가 아니다. 문서작성, 화상회의, 협업, 성과관리 등 업무지원 SaaS인지, 처리 데이터가 무엇인지, 보안성 심의와 접속통제가 있는지 확인해야 한다.

SaaS 검토의 6단계
  1. 서비스 목적이 사무관리·업무지원 범위인지 확인한다.
  2. 전자금융거래 처리 또는 고객 개인신용정보 처리 여부를 분리 판단한다.
  3. 접속 주체, 단말, 위치, 네트워크, 인증방식을 정의한다.
  4. MFA, 조건부 접근, 관리자 권한 분리, DLP, 업로드 제한, 접속기록 보관을 설계한다.
  5. 하위처리자, 데이터 저장 위치, 국외 이전, 장애 지원, 계약 해지를 확인한다.
  6. 보안성 검토 결과와 운영 로그를 정기 재검토한다.
근거: 금융위원회, 「4월 20일부터 금융회사는 내부 업무망에서 클라우드 기반 응용소프트웨어(SaaS)를 보다 원활하게 활용할 수 있습니다」, 2026, https://www.fsc.go.kr/no010101/86745; 금융보안원 「내부 업무망 SaaS 보안가이드」, https://www.fsec.or.kr/bbs/detail?bbsNo=11929&menuNo=222.

개인신용정보와 국외 이전

SaaS 활용 범위가 확대되더라도 개인신용정보, 가명정보, 국외 이전 제한 이슈는 별도로 검토해야 한다. 로드맵도 가명처리된 개인신용정보 허용 확대를 설명하면서 관련 법령상 국외 이전 제한 이슈가 남는다고 명시한다.

연구·개발망: 개발 편의와 운영망 보호를 동시에 증명

금융권 연구·개발 환경은 규제 개선 대상이다. 로드맵은 연구·개발망과 업무망 간 논리적 망분리 허용, 소스코드 등 결과물 이동 편의 확대, 가명처리된 개인신용정보 활용 허용 방향을 제시했다. 그러나 이것은 개발망을 운영망처럼 자유롭게 쓰라는 뜻이 아니다.

로드맵의 연구·개발망 보안대책 예시

연구·개발망과 전산실 간 소스코드 전송 등 제한적 통신, 인가된 서버·단말만 접속, 데이터 전송 전 악성코드 및 파일 유형 검사, 데이터 이동 내역과 예외 정책 변경의 주기적 감사, 가명정보 추가정보 분리·암호화 보관이 제시된다.

근거: 금융위원회 「금융분야 망분리 개선 로드맵」 p.9.

개발망 구축 체크리스트

  • 개발망, 테스트망, 운영망, 배포망, 소스 저장소의 경계를 분리했다.
  • 운영 데이터는 원칙적으로 반입하지 않고, 필요한 경우 가명·마스킹·샘플링 절차를 거친다.
  • 소스코드와 빌드 산출물의 운영망 반입은 승인, 해시 검증, 악성코드 검사, 서명 검증을 거친다.
  • 개발자가 운영 서버에 직접 접속하지 않고, 장애 대응 시에도 승인된 세션 기록 경로를 사용한다.
  • 오픈소스 패키지 저장소, 컨테이너 레지스트리, CI/CD SaaS 연동은 외부 접점으로 별도 관리한다.
  • 개발망 재택근무는 단말 보안, VPN, MFA, 조건부 접근, 소스 반출 금지, 화면·파일 통제를 적용한다.

AI·외부 API: 모델보다 데이터 흐름이 먼저다

생성형 AI와 외부 API는 대부분 인터넷 기반 클라우드 환경에서 제공된다. 금융위원회 로드맵은 생성형 AI 활용을 샌드박스와 보안대책 조건으로 허용하는 방향을 제시하면서, 프롬프트 차단, 클라우드 테넌트의 불필요한 외부통신 차단, 금융회사 영역과 AI 모델 영역 간 안전한 연결방식, 독점적·배타적 자원 할당, 개인정보 보호조치를 예시했다.

AI 서비스 구축 시 금지해야 할 설계
  • 업무망에서 공용 AI 웹사이트로 고객정보나 내부자료를 직접 입력한다.
  • 운영 DB에서 추출한 원문 데이터를 외부 모델 API로 전송한다.
  • AI SaaS가 입력 데이터를 모델 고도화나 제3자 제공에 사용하는 것을 계약으로 금지하지 않는다.
  • 프롬프트, 응답, 첨부파일, 검색증강 데이터에 대한 로그와 마스킹 정책이 없다.

구축 체크리스트: 착수·설계·오픈 전

착수 단계

  • 서비스가 전자금융거래, 내부업무, 고객지원, 개발·연구, 보안운영 중 어디에 해당하는지 분류했다.
  • 처리 데이터에 개인신용정보, 인증정보, 거래정보, 가명정보, 내부기밀이 있는지 표시했다.
  • 인터넷, 제휴기관, SaaS, 클라우드 API, AI API, 외주 운영자 접점이 있는지 목록화했다.
  • 적용 법령·규정·가이드와 예외 필요성을 정리했다.

설계 단계

  • 업무망, DMZ, 전산실, 관리망, 개발망, 클라우드 VPC, SaaS 경계가 도면에 표시되어 있다.
  • 서버별 인바운드·아웃바운드 통신 매트릭스가 있다.
  • 관리자 접속은 Bastion/PAM, MFA, 세션기록을 거치도록 설계했다.
  • 망간 파일 이동은 승인, 악성코드 검사, 해시 검증, 로그를 거친다.
  • 클라우드는 CSP 평가, 업무연속성, 안전성 확보조치, 계약 검토를 완료했다.
  • SaaS는 업무지원 범위, 데이터 범위, 관리자 권한, DLP, 로그 제공, 하위처리자를 확인했다.

오픈 전

  • 방화벽·보안그룹 정책이 승인서와 일치한다.
  • 관리자·서비스 계정 권한이 최소권한으로 검토되었다.
  • 로그 수집과 경보 룰이 실제 이벤트로 검증되었다.
  • 취약점 진단, 모의침투 또는 보안성 심의 지적사항이 조치되었다.
  • 운영 매뉴얼, 장애 대응, 침해사고 대응, 정책 원복 절차가 준비되었다.
근거: 전자금융거래법 제21조·제21조의2; 전자금융감독규정 제14조의2·제15조; 금융보안원 CSP 보안 관리 참고서 p.98~117.
