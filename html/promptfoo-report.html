SECTION 00OVERVIEW

promptfoo란 무엇인가

promptfoo(“프롬프트 푸”)는 LLM 애플리케이션을 체계적으로 테스트하기 위한 오픈소스 CLI이자 Node.js/Python 라이브러리다. 2024년 Ian Webster와 Michael D’Angelo가 창업했고, 출시 후 350,000명 이상의 개발자가 사용했으며 월 활성 사용자 13만 명, 포춘 500대 기업의 25% 이상이 도입한 사실상 이 분야의 표준 도구다. OpenAI와 Anthropic이 내부에서 사용해 왔다.

핵심 발상은 단순하다. 일반 소프트웨어는 입력과 출력이 결정적(deterministic)이라 단위 테스트가 쉽지만, LLM은 같은 입력에도 매번 다른 출력을 내는 확률적 시스템이다. 그래서 개발자들은 보통 프롬프트를 바꾼 뒤 두세 개 입력을 손으로 넣어 보고 “괜찮아 보이네(looks good to me)” 하고 넘어간다. promptfoo는 이 비과학적인 검수를, 코드의 단위 테스트처럼 반복 가능하고 정량적인 평가로 바꿔 준다.

promptfoo는 크게 두 가지 일을 한다. 하나는 평가(eval) — 프롬프트·모델·테스트케이스를 조합해 자동 채점하고 모델 간 성능을 벤치마킹하는 것. 다른 하나는 레드티밍(red teaming) — 적대적 입력을 자동 생성해 탈옥(jailbreak), 프롬프트 인젝션, 개인정보 유출 같은 보안 취약점을 배포 전에 스캔하는 것이다.

핵심 개념 5가지

promptfoo를 이해하는 가장 빠른 길은 평가를 구성하는 다섯 단어를 익히는 것이다. 전체 워크플로는 결국 “프롬프트 × 프로바이더 × 테스트케이스”의 매트릭스를 만들고, 각 칸을 어설션으로 채점하는 일이다.

Provider
테스트 대상 모델 또는 엔드포인트. GPT·Claude·Gemini 같은 상용 API부터 Ollama로 돌리는 로컬 모델, 직접 만든 HTTP API까지 모두 프로바이더가 된다.
Prompt
모델에 보내는 프롬프트 템플릿. {{변수}} 자리표시자로 테스트케이스마다 다른 입력을 끼워 넣는다. 여러 버전을 나란히 비교할 수 있다.
Test case
하나의 입력과, 그 출력에 기대하는 동작을 묶은 단위. 변수 값과 어설션 묶음으로 구성된다.
Assertion
출력이 통과해야 할 한 개의 규칙. “‘campus’라는 단어를 포함하라”, “30초 안에 응답하라” 같은 것. 이게 없으면 다시 “looks good to me”로 돌아간다.
Rubric
문자열 매칭으로 판정하기엔 너무 주관적인 기준(예: “이 이메일이 실제로 캐주얼하게 들리는가”)을 다른 LLM에게 평어로 채점시키는 “LLM-as-judge” 채점 지시문.

어디에 쓰는가

한 줄 요약: LLM에 “단위 테스트의 규율”을 가져오기 위해서. promptfoo는 대부분의 AI 팀이 “프로덕션에서 뭔가 터지기 전까지는 있는 줄도 모르는” 공백을 메운다. 비결정적 출력을 다루는 모델 선택, RAG 품질, 보안 — 이 세 가지 문제 영역에서 특히 강력하다.

1.1 promptfoo가 해결하는 문제

전통적 테스트는 “알려진 정답”으로 코드를 검증한다. 그러나 LLM 애플리케이션은 본질적으로 다른 도전 과제를 안고 있다. promptfoo는 이 각각에 대응한다.

문제 / 비결정성
“운 좋게 통과”를 걸러낸다

같은 프롬프트가 실행마다 다른 결과를 낸다. --repeat 3 플래그로 각 케이스를 여러 번 돌려, 두 번 통과하고 한 번 실패하는 신뢰성 문제를 한 번의 검수로는 놓치지 않게 한다.

문제 / 모델 선택
GPT vs Claude vs Llama

같은 테스트 스위트를 모든 프로바이더에 동시에 돌려 정확도·지연시간·비용을 한 표에 나란히 보여 준다. “어느 모델이 이 작업에 가장 좋은가”를 감이 아니라 데이터로 답한다.

문제 / RAG 품질
환각·맥락 충실도

검색 정확도, 답변 충실도(faithfulness), 맥락 관련성을 측정하는 전용 어설션으로 RAG 파이프라인이 “지어내지” 않는지 검증한다.

문제 / 보안
적대적 사용자

프롬프트 인젝션, 탈옥, PII 추출 시도를 레드팀 모듈이 자동으로 수백 가지 변형으로 생성해 모델이 얼마나 자주 무너지는지 점수화한다.

1.2 어설션(채점) 종류 — 무료·즉시부터 LLM 판정까지

promptfoo의 채점은 비용·속도에 따라 3개 층위로 나뉜다. 결정적(deterministic) 어설션은 공짜이고 즉각적이며, 모델 보조(model-assisted) 어설션은 다른 LLM을 호출하므로 비용이 든다. 이 계층 구조 덕분에 “싼 검사를 먼저, 비싼 LLM 판정은 꼭 필요할 때만” 설계할 수 있다.

여러 어설션은 weight로 가중치를 줄 수 있고, 테스트케이스의 threshold로 “가중 합산 점수가 X를 넘으면 통과”라는 합격선을 정한다. 예컨대 equals의 가중치를 2로 두면 contains보다 두 배 중요해진다. YAML로 정의되지만, 표현하기 어려운 로직은 JavaScript·Python 훅으로 빠져나갈 수 있다.

1.3 레드티밍 — promptfoo의 “킬러 기능”

2026년 기준 promptfoo를 가장 차별화하는 것은 레드팀 모듈이다. 구조는 세 부품이 맞물린다: 플러그인이 특정 취약점 유형의 적대적 입력을 생성하고, 전략(strategy)이 그 입력을 어떻게 전달할지(인코딩·다중 턴 등) 정하며, LLM 심판이 공격 성공 여부를 채점한다. 50개 이상의 플러그인이 6개 범주를 커버한다.

범주 1
보안 & 접근통제

프롬프트 인젝션(직접/간접), SQL·셸 인젝션, SSRF, BOLA/BFLA(객체·기능 수준 인가 결함), RBAC 우회, 디버그 접근, 시스템 프롬프트 추출.

범주 2
데이터 프라이버시

직접 PII 노출(pii:direct), 세션 간 데이터 유출(pii:session), 사회공학(pii:social), DB/API 경유 유출, COPPA·FERPA 컴플라이언스 점검.

범주 3
과도한 자율성

에이전트 고유 리스크. 의도된 범위를 넘는 행동, 권한 밖 도구 호출, 목표 하이재킹(goal hijack), 권한 상승으로 유도될 수 있는지.

범주 4
유해성 & 편향

혐오·자해 유도(harmful:hate, harmful:self-harm), 정치·종교 편향, 차별적 판단(성별·인종·지역) 등 사회·기술적 피해.

범주 5
오정보

환각(hallucination), 거짓 주장, 과의존(overreliance), 경쟁사 비방, 계약·법적 약속 위반(contracts), 사칭(imitation).

범주 6
산업별 컴플라이언스

의료·금융·보험 등 도메인 맞춤 점검과, 법무·브랜드·AI 거버넌스 팀이 정의하는 커스텀 정책 플러그인.

전달 전략(strategy) — 같은 공격, 다른 포장

플러그인이 “무엇을 묻는가”라면 전략은 “어떻게 우회하는가”다. 같은 악성 의도를 base64·rot13·leetspeak로 인코딩하거나, 다중 턴으로 서서히 끌어올리는(crescendo) 식으로 콘텐츠 필터를 뚫는다.

  • basic · prompt-injection — 페이로드를 인젝션 프레임에 감싼다.
  • jailbreak · jailbreak:composite · jailbreak:tree — DAN류 가드레일 우회, 조합·트리 탐색.
  • base64 · rot13 · leetspeak — 인코딩으로 필터를 회피.
  • multilingual — 저자원 언어(벵골어·스와힐리어·자바어 등)로 번역. 연구상 비영어권에서 안전장치가 더 약해 취약점이 잘 드러난다.
  • crescendo — 메시지마다 수위를 높이는 다중 턴 점진 공격.

프레임워크 프리셋 — 한 줄로 컴플라이언스 한 벌

개별 플러그인을 일일이 고르지 않고, 표준 프레임워크를 한 줄로 켤 수 있다. 웹 UI는 그 결과를 OWASP·NIST·MITRE 통제 항목에 매핑한 “컴플라이언스 성적표”로 출력한다 — 심각도 등급과 증거 추적까지 포함해 비개발자도 읽을 수 있다.

  plugins:
    - owasp:llm          # OWASP LLM Top 10 전체
    - pii
    - ssrf
    - excessive-agency
  strategies:

현업에서 가장 크게 드러나는 강점: promptfoo의 엔진은 “임의 입력 × 임의 프로바이더(코드) → 어설션 채점 → pass/fail·0~1 점수 + 머신리더블 리포트”를 반복적으로 돌리는 것이다. 이 세 축 중 어느 것도 LLM에 묶여 있지 않다. 즉 promptfoo는 LLM 전용 도구가 아니라, 아무 데이터·아무 코드를 스크립트 베이스로 평가하는 범용 테스트/회귀 하니스이며, 일반 ML에도 그대로 쓸 수 있다.

한 문장 재정의

“프롬프트 × 프로바이더 × 테스트케이스” 매트릭스에서 프롬프트를 ‘입력 데이터’로, 프로바이더를 ‘실행할 코드’로, 어설션을 ‘pass/fail·확률값 산출 함수’로 일반화하면, LLM 평가는 이 범용 엔진의 한 가지 특수 사례일 뿐이다. parquet·json·jsonl·csv 어떤 데이터든 → python·CLI·API 어떤 코드든 → 결국 pass/fail 또는 0~1 점수로 환원해 반복 산출한다.

1B.1 세 축 모두 “LLM 무관”으로 일반화된다

축 1 · 프로바이더
실행 대상 = 임의 코드

모델 API가 아니라 아무 코드/스크립트여도 된다. Python provider(call_api), exec/스크립트 provider(임의 실행파일), JS·Go provider, HTTP API. 분류기·피처 파이프라인·CLI 바이너리·마이크로서비스가 모두 “테스트 대상”이 된다.

축 2 · 데이터
입력 = 임의 데이터셋

CSV·JSON·JSONL·YAML·Excel(XLSX)이 네이티브. HuggingFace·Google Sheets·Azure Blob도. parquet 등은 Python/JS 테스트 생성기로 pandas·pyarrow로 읽어 케이스를 만들면 그대로 흘러든다. 벡터 DB에서 vars를 끌어올 수도 있다.

축 3 · 어설션
판정 = pass/fail·0~1 점수

Python/JS 어설션은 bool(통과/실패)·float(0~1 점수)·GradingResult를 반환한다. threshold로 점수 합격선을 정하고, derived metric으로 F1·정밀도·재현율·MAPE 같은 집계 지표까지 산출한다.

1B.2 결과는 반복 가능하고 기계가 읽는다

이게 “스크립트 베이스로 반복 산출”의 핵심이다. 동일 설정을 몇 번 돌려도 같은 매트릭스가 나오고, 결과는 사람이 보는 HTML뿐 아니라 JSON·CSV·JSONL·JUnit XML로 빠진다 — 즉 GitLab·Jenkins·Azure Pipelines 같은 일반 CI 리포트로 바로 들어간다. 비결정 신뢰성은 --repeat로, 비용·속도는 캐싱으로 관리한다.

결정적 기능 — standalone 어설션 모드

이미 산출해 둔 출력(예: ML 모델의 예측 결과)을 output.json 배열로 넣고 프로바이더 재실행 없이 어설션만 돌릴 수 있다. 즉 promptfoo를 “채점·집계·리포팅 레이어”로만 떼어 써서, 외부에서 돌린 임의 파이프라인의 결과를 pass/fail·점수·CI 리포트로 환원하는 용도가 된다.

2.3 머리맞대기 — promptfoo vs 주요 경쟁자

vs DeepEval — “YAML CLI” vs “Python pytest”

가장 흔한 양자택일이다. 둘은 서로 다른 실패 부류를 덮어, 실무에선 같이 쓰는 게 표준 패턴이다.

  • promptfoo가 이기는 곳 — 레드티밍·적대적 테스트·멀티모델 프롬프트 비교. YAML CLI와 40개 이상의 레드팀 플러그인으로 “LLM 앱을 체계적으로 부수는” 데 가장 강하다. 노드(Node.js) 환경 팀에 자연스럽다.
  • DeepEval이 이기는 곳 — Python 네이티브 팀의 pytest 기반 CI 품질 게이트. pytest tests/가 LLM 평가를 일반 테스트와 함께 돌린다. 14~60+ 메트릭(faithfulness·hallucination·bias·toxicity·tool-correctness 등), G-Eval 커스텀 메트릭, MMLU·TruthfulQA 같은 표준 벤치마크 내장.
  • 표준 조합 — promptfoo를 레드팀 CI 게이트로, DeepEval을 메트릭 CI 게이트로. 메트릭 폭은 DeepEval(60+), 보안·레드팀 깊이는 promptfoo가 우위. promptfoo의 YAML은 “선언적이라 쉽다”는 강점인 동시에 “플러그인 추상화가 경직돼 보인다”는 비판도 받는다.

vs RAGAS — 겹치지 않는다

RAGAS는 RAG 평가에 한정된 학술 메트릭(faithfulness, context precision/recall, answer similarity 등)에 특화돼 있고, 레드티밍이나 비-RAG 평가 기능이 없다. 따라서 promptfoo와 경쟁이라기보다 보완재다. RAG가 핵심이면 RAGAS로 검색·생성을 깊게 측정하고, 보안·멀티모델은 promptfoo로 덮는 식이다.

vs LangSmith / Langfuse — 평가 vs 관측

이쪽은 “테스트 도구”가 아니라 “관측 플랫폼”이다. LangSmith는 LangChain/LangGraph 스택에 깊게 통합돼 트레이스를 자동 수집하지만 호스티드 SaaS이고 시트당 과금이라 혼합 스택 팀에는 마찰이 있다. Langfuse는 오픈소스·셀프호스트가 가능해 데이터 거주성·비용에서 유리하지만 평가 깊이는 전용 도구보다 얕다. promptfoo의 약점이 정확히 여기다 — 결과가 로컬에 머무르며 중앙집중 실험 추적이 없다. 그래서 “promptfoo로 평가/레드팀 + Langfuse·Phoenix로 트레이싱”이 자주 권장된다.

vs Garak · PyRIT — 레드팀 순수주의

레드티밍만 떼어 비교하면 promptfoo는 “50+ 플러그인 + YAML CI 통합”의 제너럴리스트다. PyRIT(Microsoft, 자사 레드팀에 내부 사용)는 프로그래매틱 캠페인에 더 세밀한 제어를 주는 Python 프레임워크고, Garak(NVIDIA 후원), Giskard(RAG 강점)도 벤더 독립 오픈소스 대안이다. 이 “벤더 독립성”이 다음 절의 핵심 쟁점이 된다.

인수(2026.03.09) 후 2개월간 제품 자체는 눈에 띄게 나빠지지 않았다 — CLI는 그대로 돌고, YAML 스키마도 불변, 공격 라이브러리도 계속 갱신됐다. 그러나 보안 감사 맥락에서 새 질문이 생겼다: “OpenAI 배포를 감사하려는 기업이, 평가받는 주체(OpenAI)가 소유한 채점기를 신뢰할 수 있는가?”

  • 업계 consolidation 신호 — 2개 분기 만의 두 번째 대형 AI 레드팀 인수다(2025.11 Splx→Zscaler, 2026.03 promptfoo→OpenAI). 구조적으로 독립적인 벤더가 줄고 있다.
  • 독립 대안 — 채점기 독립성이 감사 요건이라면, 오픈소스 Garak·PyRIT·Giskard는 프런티어 모델 소유주가 없다는 점에서 구조적 중립성을 갖는다. 상용 쪽에는 독립 플랫폼들이 있다.

요점: 일상적 평가·CI 회귀·프롬프트 비교에는 promptfoo가 여전히 탁월하다. 다만 “OpenAI 모델을 감사하는 보안 딜리버러블”이 목적이라면 채점기 소유 구조를 의식하고 벤더 독립 도구를 병행·검토하는 게 합리적이다.

2B.1 OpenAI는 왜 굳이 샀나

핵심은 산 게 “promptfoo CLI”가 아니라 OpenAI Frontier를 위한 신뢰라는 점이다. Frontier는 2026년 2월 출시된 엔터프라이즈 AI 에이전트(코워커) 플랫폼으로 Uber·State Farm·Intuit·Thermo Fisher 등을 고객으로 둔다. 에이전트가 실제 업무 인프라에 닿는 순간, 평가·보안·컴플라이언스는 “있으면 좋은 것”이 아니라 배포의 전제 조건이 된다. OpenAI 스스로 이를 인수 사유로 명시했다.

이유 1 · 플랫폼 공백
“데모 → 프로덕션” 관문

이유 2 · Buy > Build
못 만드는 건 “신뢰”

레드팀 기능은 만들 수 있어도 채택과 신뢰는 못 만든다. 포춘 500의 25%·개발자 35만이 이미 쓰는 “검증된 업계 표준”이라는 점이 규제 조달에서 결정적이다.

이유 3 · Acqui-hire

OWASP·NIST·MITRE에 매핑된 공격 라이브러리(몇 년 걸려 쌓이는 자산)와 검증된 팀(ex-Discord·ex-Smile Identity, 23명)을 한 번에 확보했다.

이유 4 · 경쟁 타이밍
에이전트 책임성 경쟁

같은 주 Codex Security(앱 보안 에이전트)를 내놓았고, 2026년 2월 Anthropic의 Claude Code Security가 같은 영역을 겨눴다. “에이전트 책임성 = 새 프로덕션 관문”이 굳어지던 흐름이다.

두 층위로 봐야 한다. promptfoo를 “시장에서 가치 있게” 만든 pain과, “OpenAI에게 특별히 절실했던” pain은 다르다.

2B.3 우리에겐 무슨 의미인가 — 무엇이 바뀌나

우리 상황(오프라인·에어갭 + 범용 ML/LLM 하니스로 사내 사용)에 한정해 솔직하게 말하면:

지금 당장: 운영상 변화 없음 (그게 핵심)

MIT 라이선스와 공개 저장소가 유지되고, 우리가 이미 vendoring한 버전·YAML 스위트·Claude Code(MCP) 내부 워크플로는 그대로 돈다. 라이선스는 이미 확보한 코드와 포크할 권리를 보호하므로 소유주가 누구든 우리 손의 코드는 회수되지 않는다. 특히 에어갭 + 범용 하니스 용법은 promptfoo를 쓰는 방식 중 인수 영향이 가장 적다 — 우리는 이미 호스티드 의존을 끊었고, OpenAI가 가져간 가치(Frontier의 호스티드 엔터프라이즈 레드팀)는 애초에 우리 경로 밖이기 때문이다.

진짜 바뀌는 것 — “비트”가 아니라 “궤적과 맥락”

변화 1 · 로드맵
OpenAI 우선순위로 정렬

유지보수 방향이 Frontier 우선순위(에이전트 보안·OpenAI 모델)에 맞춰진다. 비-OpenAI 모델 커버리지·범용 ML 용도는 후순위가 될 수 있다. 멀티 모델 지원 약속은 상용 로드맵과 충돌하는 첫 결정에서 시험대에 오른다.

변화 2 · 중립성
채점기 ↔ 피채점자

향후 외부 감사·규제 대응으로 OpenAI 모델 자체를 평가한다면 “평가받는 당사자가 채점기를 소유”하는 이해상충이 생긴다. 단, 우리 내부 에어갭 용도엔 사실상 무관(OpenAI를 감사하지도, 호스티드를 호출하지도 않음).

그래서 구체적으로 할 일 (가벼움)

  1. 버전 고정(freeze). 이미 컨테이너에 vendoring했으니 “프리즈된 의존성”으로 다루고, changelog 검토 후에만 의도적으로 올린다. 강제 변경에 휘둘릴 일이 없다.
  2. 범용 하니스 용법을 중심에 둔다. “임의 데이터 + 임의 코드 → pass/fail·점수”는 구조적으로 벤더 중립이라 로드맵 드리프트에 가장 덜 노출된다.
  3. 포크 옵션을 주머니에 둔다. MIT라 포크가 가능하고 기술적으로 어렵지 않으며 커뮤니티 포크 논의도 이미 있다. 당장 할 필요는 없고, 사용을 portable하게(YAML + 로컬 프로바이더 — 이미 그렇다) 유지하면 된다.
  4. 2개 신호를 ~12개월 관찰. (a) 비-OpenAI 모델 커버리지가 계속 1급으로 업데이트되나, (b) OSS 저장소가 활발히 유지되나 vs Frontier 전용으로 표류하나. 둘 다 악화되면 그때 벤더 중립 대안(레드팀: Garak·PyRIT·Giskard / 메트릭: DeepEval)을 검토.
  5. 용도를 분리한다. 중립성이 조달 이슈가 아닌 내부 평가·CI·회귀는 promptfoo를 그대로, 규제 데이터·외부 감사가 걸리는 보안·컴플라이언스 레이어는 벤더 중립 도구로 분리.
전략 결론

사내 오프라인 범용 하니스로서는 저위험이고 우리 포지션은 좋다. 달라진 단 하나는 인식의 전환이다 — promptfoo를 “중립적 커뮤니티 프로젝트”가 아니라 “우리가 로드맵을 통제하지 않는 vendored 도구”로 취급하고, 그에 맞게 핀(pin)·포크(fork) 전략을 갖추는 것. 우리가 이미 한 에어갭·vendoring·범용화는 사실상 그 대비를 선제적으로 끝내 둔 셈이다.

이 절의 접근은 실제 배포로 확인되었다: 오프라인으로 컨테이너를 빌드하고 충분한 문서를 함께 제공해 Claude Code CLI가 내부에서 평가를 구동하도록 했고, 에어갭 환경용으로 리빌드해 격리망 내부에서 그대로 사용 중이다. 참고로 promptfoo는 promptfoo mcp로 MCP 서버를 띄워 자신의 eval·테스트 기능을 도구로 노출할 수 있어, Cursor·Claude Desktop·Claude Code 같은 에이전트가 내부에서 평가를 호출하는 워크플로에 자연스럽게 들어맞는다.

먼저 알아야 할 진실

promptfoo 공식 문서는 분명히 한다: 텔레메트리·업데이트 차단 같은 환경변수 설정은 “네트워크 방화벽이 아니다.” 진짜 엄격한 오프라인/에어갭이 필요하면 (로컬 프로바이더) + (네트워크 egress 통제) + (필요 시 Enterprise On-Prem)를 조합해야 한다. 즉 promptfoo 설정 + OS/방화벽 차단이 함께 가야 한다.

3.1 1단계 — 로컬 프로바이더로 모델을 가져온다

외부 API 대신 로컬에서 추론하는 프로바이더를 쓰면 promptfoo가 외부 모델 API에 닿을 일이 없다. 대표 선택지는 Ollama(가장 흔함), LocalAI(OpenAI 호환 셀프호스트), 그리고 vLLM·llama.cpp 같은 OpenAI 호환 엔드포인트다.

Ollama 연결

Ollama는 모델을 한 번 pull로 받아 두면 인터넷 없이 추론·임베딩이 모두 로컬에서 돈다. promptfoo는 /api/generate와 /api/embeddings를 모두 지원한다 — 즉 llm-rubric 채점과 similar(유사도) 어설션까지 로컬 모델로 처리할 수 있다.

terminalbash
# 인터넷 연결된 상태에서 모델 미리 받기
ollama pull llama3.3
ollama pull nomic-embed-text     # 임베딩(유사도 어설션용)

# promptfoo가 IPv4로 직접 닿게(localhost IPv6 이슈 회피)
export OLLAMA_BASE_URL="http://127.0.0.1:11434"

# 다른 머신의 Ollama에 붙기(사내 GPU 서버 등)
export OLLAMA_BASE_URL="http://192.168.1.100:11434"
# 이때 서버는 OLLAMA_HOST=0.0.0.0:11434 로 떠 있어야 함
promptfooconfig.yamllocal-only
providers:
  - ollama:chat:llama3.3            # 대상 모델
  - ollama:completion:mistral

# 로컬 모델을 "채점기"로도 사용 → LLM 호출이 외부로 안 나감
defaultTest:
  options:
    provider:
      text: ollama:chat:llama3.3          # llm-rubric 채점기
      embedding: ollama:embeddings:nomic-embed-text  # similar 채점기
리소스 절약 팁

기본적으로 promptfoo는 모든 프로바이더를 동시(concurrent)에 평가한다. -j 1 플래그를 주면 한 번에 모델 하나·프롬프트 하나씩 순차 실행해, 한 모델만 메모리에 올라가 로컬 GPU/RAM을 아낀다. 큰 로컬 모델을 돌릴 때 유용하다.

LocalAI / vLLM / llama.cpp

OpenAI 호환 API를 제공하는 엔진이면 promptfoo의 OpenAI 호환 프로바이더 설정으로 baseURL만 로컬 주소로 바꿔 붙일 수 있다. 사내 게이트웨이(http://internal-llm:8000/v1)도 동일하게 처리된다.

3.2 2단계 — 외부로 나가는 부수 호출을 끈다

로컬 모델만 써도 promptfoo는 기본적으로 텔레메트리, 업데이트 확인, 그리고 일부 레드팀 기능의 원격 생성을 시도할 수 있다. 아래 환경변수로 이를 차단한다. 텔레메트리에는 프롬프트·모델 출력·테스트케이스·API 키·전체 설정 파일이 포함되지 않는다(패키지 버전·CI 여부 등 메타데이터뿐)지만, 오프라인에선 모두 꺼 두는 게 깔끔하다.

offline.envenv vars
# 익명 사용 분석(텔레메트리) 비활성화
export PROMPTFOO_DISABLE_TELEMETRY=1

# NPM 레지스트리 업데이트 확인 비활성화
export PROMPTFOO_DISABLE_UPDATE=1

# 레드팀: 원격 생성 폴백 비활성화(자체 OPENAI_API_KEY나 로컬 모델 필요)
export PROMPTFOO_DISABLE_REDTEAM_REMOTE_GENERATION=true

# 사내 CA 인증서가 있는 환경(기업 프록시)
export PROMPTFOO_CA_CERT_PATH=/path/to/ca-bundle.pem
레드팀의 결정적 제약 — 원격 전용 기능

이게 오프라인에서 가장 중요한 함정이다. promptfoo의 일부 레드팀 플러그인·전략은 promptfoo의 호스티드 추론을 요구하며 로컬에서 실행할 수 없다(문서에 🌐로 표시됨). 즉, 완전 에어갭에서는 이 고급 공격 기법들을 쓸 수 없다.

원격 전용 전략 예: audio · citation · gcg · goat · jailbreak:composite · jailbreak:hydra · jailbreak:likert · jailbreak:meta. 또한 promptfoo redteam poison은 자체 API 키가 있어도 항상 호스티드 생성을 쓴다. 임베딩·moderation 어설션도 별도 프로바이더 지정이 필요하다.

정리하면, 에어갭에서 가능한 것은 평가(eval) 전반 + 로컬 모델 채점 + 직접 정의한(static) 레드팀 케이스이고, 적대적 입력을 동적으로 생성하는 일부 고급 전략은 제약된다. 완전 격리 환경에서 풀 레드팀이 필요하면 Enterprise On-Prem 라인을 검토해야 한다.

3.3 3단계 — 결과 저장은 이미 로컬(SQLite)

기본적으로 평가 결과는 로컬에 저장된다. CLI는 SQLite DB(promptfoo.db)를 사용자 홈의 .promptfoo 디렉터리에 둔다. 공유(sharing)·클라우드 동기화 같은 클라우드 기능을 명시적으로 호출하지 않는 한 결과가 밖으로 나가지 않는다. 즉 별도 조치 없이도 데이터는 로컬에 머문다.

3.4 (선택) 웹 UI를 셀프호스팅 — Docker

팀이 평가 이력을 웹 UI로 보고 싶다면 공식 Docker 이미지로 서버를 띄울 수 있다. 단, 공식 문서는 이 셀프호스트 버전을 프로덕션 권장하지 않는다(SQLite 단일 파일, 수동 영속화, 멀티팀·RBAC·수평확장 없음 — 복제본 2개 이상이면 “Job not found” 오류). 엔터프라이즈 고객은 이 이미지 대신 별도 엔터프라이즈 이미지를 받아야 한다.

UI는 http://localhost:3000. CLI에서 이 셀프호스트 인스턴스로 결과를 올리려면 두 변수를 가리키게 한 뒤 명시적으로 업로드한다:

# 데이터·blob 영속 경로 / 설정 디렉터리 재정의

# PROMPTFOO_CONFIG_DIR 로 기본 설정 디렉터리 변경 가능

3.5 진짜 에어갭(air-gapped) 운영 절차

정부·금융·의료·국방처럼 “네트워크를 떠나는 데이터 = 보안 사고”인 환경에서는 promptfoo 설정만으로 부족하다. 로컬 AI 생태계(Ollama·LocalAI·llama.cpp)는 에어갭에서 잘 돌지만, 핵심은 운영 규율이다. 권장 절차:

  1. 스니커넷으로 모델 반입 — 연결된 머신에서 Ollama 설치본과 ~/.ollama/models/ 디렉터리(매니페스트+blob)를 통째로 외장 매체에 복사해 격리 머신의 동일 경로로 옮긴다. ollama list로 인식 확인. 레지스트리 pull은 에어갭에서 동작하지 않으므로 파일 직접 복사가 정석이다.
  2. 오프라인 패키지 반입 — promptfoo는 npm/pip 패키지이므로, 동일하게 오프라인 미러나 vendored tarball로 설치한다. Node.js 런타임도 함께.
  3. egress 차단을 OS/방화벽으로 강제 — promptfoo 환경변수는 방화벽이 아니다. 프로세스 단위 아웃바운드 차단 규칙으로 “설정 실수로라도 밖으로 못 나가게” 강제한다. 예: macOS pf, Linux iptables/nftables, Windows 방화벽 아웃바운드 규칙.
  4. 로컬 채점기 고정 — llm-rubric·similar 어설션의 채점/임베딩 프로바이더를 모두 로컬 Ollama로 명시(3.1의 options.provider). 누락 시 기본 채점기가 외부를 시도할 수 있다.
  5. 원격 전용 레드팀 기능 회피 — 🌐 표시 전략·플러그인은 제외하고, 정적 레드팀 케이스(직접 작성하거나 HuggingFace 데이터셋에서 반입)로 대체한다. 풀 동적 레드팀이 필수면 Enterprise On-Prem.
  6. 검증 — Little Snitch/LuLu(macOS)나 네트워크 모니터로 평가 실행 중 promptfoo·ollama 프로세스의 아웃바운드 연결이 0인지 확인. 인터넷을 물리적으로 끊고 동일 평가가 정상 동작하면 오프라인 확정.

4.2 안티패턴 — 흔히 하는 실수

  • 프롬프트만 테스트하기 — 프롬프트를 격리해 검증하면 동적으로 프롬프트가 생성되는 실제 서비스 동작을 못 본다. “테스트 대상(system under test)”은 프롬프트가 아니라 서비스 자체여야 한다. HTTP 타깃으로 실제 엔드포인트를 친다.
  • 한 번 돌리고 끝 — 비결정성 때문에 단발 통과는 “운 좋게 통과”일 수 있다. --repeat로 반복 신뢰성을 본다.
  • LLM 채점 남발 — 모든 어설션을 llm-rubric으로 하면 느리고 비싸다. 결정적 검사를 먼저 깔고, 주관적 기준만 LLM에 맡긴다.
  • 오프라인인데 채점기 방치 — 대상 모델만 로컬로 바꾸고 채점기/임베딩 프로바이더를 안 지정하면 기본 채점기가 외부를 시도한다. 둘 다 로컬로 고정해야 진짜 오프라인.
  • 레드팀을 프로덕션 모니터로 오해 — promptfoo는 배포 전 도구다. 라이브 트래픽 감시 용도가 아니다.

4.3 FAQ

코딩 없이 쓸 수 있나?
대부분의 일반 시나리오는 YAML만으로 가능하다. 커스텀 어설션·고급 로직이 필요할 때만 JavaScript/Python 훅을 쓴다.
어떤 모델을 지원하나?
OpenAI·Anthropic·Google·Azure·AWS Bedrock·HuggingFace, Ollama 등 로컬 모델, 그리고 OpenAI 호환 API를 가진 모든 프로바이더(50+). 같은 프롬프트를 전부에 동시 비교할 수 있다.
데이터가 외부로 나가나?
기본은 아니오. 러너는 로컬에서 돌고 설정한 프로바이더로 직접 호출된다. 공유·클라우드 동기화·호스티드 리포트·일부 레드팀 원격 생성 같은 클라우드 기능을 명시적으로 켤 때만 데이터가 promptfoo 운영 서비스에 닿는다.
API 키가 텔레메트리로 새나?
아니오. 키는 로컬 환경변수에서 읽고 그 자체로 전송되지 않는다. 다만 설정·요청 예시·레드팀 셋업 폼에 직접 붙여 넣은 비밀값은 로컬 아티팩트나 클라우드 기능에 저장·전송될 수 있으니 그런 폼에 실제 시크릿을 넣지 말 것.
CI/CD에 넣을 수 있나?
그렇다. 머신리더블 출력(JSON 등)을 내므로 GitHub Actions·GitLab CI·Jenkins·CircleCI 어디든 붙는다. 통과율이 임계값 밑이거나 취약점 점수가 높으면 빌드를 실패시킨다.
인수 후에도 안전하게 쓸 수 있나?
오픈소스 CLI는 MIT로 유지되고 저장소도 공개다. 인수 2개월 시점까지 제품은 눈에 띄게 나빠지지 않았다. 단 보안 감사에서 채점기 독립성이 요건이면 벤더 독립 대안(Garak·PyRIT·Giskard)을 병행 검토.
비용은?
코어는 무료(MIT). 클라우드 Team $50/월, Enterprise 커스텀. 평가 중 호출한 LLM 토큰비는 별도이며 로컬 모델을 쓰면 0이 된다.
한 문장 결론

promptfoo는 “LLM 앱에 단위 테스트와 침투 테스트의 규율을 동시에 가져오는, 로컬 우선 오픈소스 도구”다. 프롬프트·모델 선택과 보안 레드티밍이 목적이고 Node 환경이라면 2026년에도 가장 빠른 출발점이며, 메트릭 깊이(DeepEval)나 프로덕션 관측(Langfuse·Phoenix)은 별도 도구로 보완하는 2-스택 구성이 정석이다.
